# Proposal template — technical writing / documentation

Use for: READMEs, API docs, setup guides, how-tos, internal process docs.
NOT for blog/content/SEO writing — that category is rejected at triage.
Fill from the listing; leave `[[OPERATOR: ...]]` slots for the operator.
Keep the final proposal under ~150 words.

---

Hi [[OPERATOR: client first name if shown]],

You need {one-sentence restatement: what doc, for which reader — "docs your
users can follow without filing a support ticket", etc.}.

What I'd deliver:

- {The concrete doc: "a README covering install, config, and the three main
  workflows" / "API reference for your N endpoints with request/response
  examples"}
- Written for {the actual audience from the listing: end users / developers /
  new hires} — I'll follow your product hands-on where possible, so the steps
  are verified, not guessed
- In {their format: Markdown / Google Doc / their docs site's format}

I work with modern AI-assisted tooling, so first draft lands in {24–48h},
and every instruction is checked before you see it.

One question: {sharpest scope question — usually: what access/materials can
they share (repo, staging login-free demo, existing drafts)?}

[[OPERATOR: one personal line — why their product/domain interests you.]]

— [[OPERATOR: name]]

---

**Acceptance checklist to include in delivery** (edit per job): every
step-by-step actually executed or explicitly marked as unverifiable; code
samples run as written; terminology matches the client's product UI; format
renders correctly where it will live.
